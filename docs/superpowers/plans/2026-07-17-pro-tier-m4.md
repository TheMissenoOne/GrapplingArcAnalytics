# Pro Tier M4 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add the tracked Pro entitlement/schema contract, deterministic batch payload generation, and safe CI without enabling production scheduling.

**Architecture:** Analytics owns the PostgreSQL tables, RLS setup, payload validation, and a manual batch publisher. The App consumes only versioned JSON through Supabase and retains its offline-first fallback. A normal CI workflow runs entirely offline; the credentialed publisher remains manual until the App validates M5 end-to-end.

**Tech Stack:** Python 3.12, SQLAlchemy/Alembic, pytest, Ruff, GitHub Actions, Supabase/Postgres.

## Global Constraints

- One Analytics PR and one paired App PR; never a shared branch.
- No live DB mutation by an agent; migration application and RLS probes are human handoff.
- `profiles.is_pro` is server-admin-owned: authenticated users must not update that column.
- Guest/offline App users remain free and make no Supabase request.
- Snapshot/dossier payloads are schema-versioned; no raw ELO comparison crosses App/pro-athlete domains.
- Batch publishing stays manual until M5 accepts the payload contract.
